요청하신 IGS(IGES) 파일 구조
현재 프로젝트가 “포인트(엔터티 116)”를 읽어 스프링 축/좌표 변환에 쓰고 있으므로, 그 부분에 특히 초점을 맞춰 설명합니다.

## IGES 개요
- 형식: ASCII 기반 80자 고정 길이 레코드(줄)로 구성된 교환 포맷
- 확장자: .igs, .iges
- 목적: CAD 시스템 간 기하/속성 데이터를 교환

## 파일은 5개 섹션으로 구성
각 줄의 73열(1-indexed)에 섹션 문자(S/G/D/P/T)가 들어갑니다. 1~72열이 데이터, 74~80열은 7자리 시퀀스 번호.

1) Start Section (S)
- 자유 텍스트 설명. 파일 목적/생성기 등.
- 파싱 필요 없음.

2) Global Section (G)
- 파서가 “파일 전체의 해석 방식”을 알아내는 핵심 메타데이터.
- 주요 항목:
  - 파라미터 구분자/레코드 구분자(보통 , 와 ;). 실제 파일에선 Global 섹션 첫 줄에서 “,;” 형태로 정의되는 경우가 많습니다.
  - 수치 포맷(부동소수 자릿수 등), 단위(인치/밀리미터), 저자/시스템, 모델 스페이스 크기(최소/최대 좌표) 등.
- 중요: 여기서 얻은 “구분자”로 P 섹션을 토큰화해야 합니다. 하드코딩된 콤마/세미콜론 전제는 일부 파일에선 실패합니다.

3) Directory Entry Section (D)
- 모든 “엔터티”의 메타 정보. 각 엔터티는 80자 레코드 2줄(=DE 1개)을 차지.
- 2줄 합쳐 20개의 8자 필드(총 160자). 주요 필드(대표적, 시스템마다 일부 다름):
  - (1) Entity Type Number: 엔터티 종류(예: 116=Point, 110=Line, 100=Circular Arc, 126=Rational B‑Spline Curve)
  - (2) Parameter Data Start Pointer: 해당 엔터티의 파라미터 데이터가 P 섹션 몇 번째 레코드에서 시작하는지
  - (3) Structure, (5) Level(레이어), (8) Transformation Matrix Pointer(124 엔터티 참조), (10) Status Number(가시/선택/분리 등 플래그)
  - (13) Line Weight, (14) Color(314 Color Definition 참조 가능), (15) Parameter Line Count, (16) Form Number(같은 타입의 변형)
  - (19) Entity Label, (20) Subscript Number
- 파싱 흐름: D 섹션을 순회하며 “관심 있는 타입(예: 116)”만 골라 P 섹션 위치와 보조정보를 기록.

4) Parameter Data Section (P)
- 실제 기하 파라미터가 Global 섹션에서 정의한 구분자(보통 ,)로 나열, 엔터티 끝은 레코드 구분자(보통 ;)로 종료.
- 하나의 엔터티가 여러 레코드(여러 줄)를 차지할 수 있음. D 섹션의 “Parameter Line Count”로 묶어 읽습니다.
- 숫자 표현은 지수형(예: 1.23D+02, 4.56E-03)도 허용. 파싱 시 float 변환에서 E/D 지수 모두 처리 필요.
- 예시(포인트; 엔터티 116):
  - 표준: “116, X, Y, Z;”
  - 일부 CAD/내보내기: “116,X,Y,Z,0;”처럼 vendor 확장/더미 필드가 붙기도 함(표준 필드 3개만 쓰는 것이 IGES 정의).

5) Terminate Section (T)
- 파일 요약/검증용 레코드 수 집계(각 섹션 줄 수).
- 툴이 파일 일관성/손상을 검증할 때 사용.

## 엔터티(Entities) 간단 지도
- 116: Point (X,Y,Z)
- 110: Line
- 100: Circular Arc
- 126: Rational B‑Spline Curve (NURBS 곡선)
- 128: Rational B‑Spline Surface (NURBS 서피스)
- 124: Transformation Matrix (DE의 transform pointer로 참조)
- 314: Color Definition
- 406: Property (Level, Name 등 다양한 속성 집합)
- 402: Group/Associativity

## 좌표/단위/변환
- 단위: Global 섹션의 “Units Flag/Name”에서 결정(1=inches, 2=mm 등; 구현에 따라 name이 우선될 수 있음).
- 좌표계: 일반적으로 우수(right-handed) 3D 직교 좌표. 특정 시스템별 회전/변환은 124 변환행렬 엔터티로 연결.
- 파싱 시: DE의 Transformation Matrix Pointer(필드 8)가 124 엔터티를 가리키면, 파라미터 좌표에 해당 행렬을 적용해야 “월드 좌표”가 됨(단, 많은 파일은 단순/영행렬이거나 생략).

## 본 프로젝트와의 연결 (포인트 116)
- 현재 코드의 정규식:
  - r'116,(-?…x),(-?…y),(-?…z),0;' 형태를 읽습니다.
  - 주의:
    - 지수 표기(E/e/D) 값, 공백, 탭, 줄바꿈 분할, 지역적 소수점 표기(‘,’ vs ‘.’) 등 변형에 취약할 수 있음
    - Global 섹션의 파라미터 구분자가 ‘,’가 아닐 수도 있음(드묾)
    - 벤더 확장으로 116 뒤에 4번째 파라미터(0)가 덧붙는 케이스가 있으나 표준은 X,Y,Z 3개입니다
- 더 견고한 파싱 절차(권장):
  1) Global 섹션에서 실제 구분자/레코드 구분자/단위를 파악
  2) D 섹션을 순회해 116 엔터티만 수집, P 섹션 포인터와 parameter line count로 묶어 읽기
  3) P 섹션 토큰을 구분자 기반으로 분리하고, E/D 지수 포맷 모두 허용하여 float 변환
  4) 필요 시 124 변환행렬을 따라 좌표에 변환 적용
  5) 단위 mm로 정규화(필요 시 스케일)
- 실무 팁:
  - IGES 줄은 80자 고정이지만, P 섹션 파라미터는 엔터티 끝의 레코드 구분자(;)까지만 논리적으로 읽으면 됩니다(줄 바꿈 가능).
  - Directory Entry의 Status Number 비트(10번째 필드)로 가시/선택성 등을 판단 가능(보통 좌표 추출에는 영향 적음).

## 파싱 시 자주 겪는 함정
- 과도한 정규식 의존
  - 숫자 형식(지수), 공백, 구분자 변형에 약함 → Global 기반 토크나이저가 안전.
- 여러 레코드에 걸친 P 데이터
  - 한 엔터티가 줄을 여러 개 차지. D의 “parameter line count” 또는 “다음 엔터티 시작”을 기준으로 묶기.
- 변환행렬(124) 미적용
  - 일부 파일에선 실제 좌표가 transform을 거쳐야 실제 위치가 됩니다.
- 단위 변환 누락
  - Global이 inch일 때 mm 기대하면 치수가 25.4배 빗나감.

## 최소 구현 예(요약 절차)
- S/G/D/P/T 섹션 분리 → G에서 delim/unit 확인
- D 전체 스캔 → type 116만 남김
- 각 116 DE의 P 포인터로 P 엔터티 읽기 → X,Y,Z 파싱
- 필요 시 124 변환 적용 → 통일 단위(mm)로 스케일
- 좌표 배열 반환

## 현재 코드 개선 제안(옵션)
- 정규식 → 섹션 기반 파서로 확장(특히 P 토큰화와 지수 파싱)
- Global 섹션에서 단위/구분자 자동 인식
- 124 변환행렬 처리(있을 경우)
- 숫자 파서: ‘E/e/D’ 지수, 앞뒤 공백, ‘+/-’ 부호 안정 처리

이 구조를 이해해 두면, 포인트 외에도 NURBS(126/128)를 직접 읽어 스플라인/서피스로 곧장 적합하거나, 색/레이어(314/406)/트랜스폼(124)까지 일관되게 적용할 수 있습니다.
